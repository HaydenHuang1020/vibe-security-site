VS-01

Cross-tenant order read through server route

High · Confirmed

Trust boundary

The authenticated user boundary held, but the tenant boundary did not. The server accepted an order identifier from the client without deriving tenant ownership from the session.

Reproduction evidence

  1. Tenant B created synthetic order ord_test_b_1042 in staging.
  2. Tenant A requested GET /api/orders/ord_test_b_1042 using its own valid session.
  3. The route returned 200 with Tenant B's synthetic email and order amount. Expected result: 403 or no record.

Code evidence

const order = await db.order.findUnique({ where: { id } });
return Response.json(order); // tenant ownership not enforced

Illustrative code for this fictional report. A real delivery cites the actual repository path, line, request, response, and timestamp.

Fix guidance

Resolve membership from the authenticated server-side identity and query by both order ID and authorized tenant ID. Enforce the same condition in database policy so a future route cannot bypass it.

const tenantId = await requireTenantMembership(session.user.id);
const order = await db.order.findFirst({ where: { id, tenantId } });
04 / Re-test

  • PENDING Stripe webhook replay test awaits the customer's next staging deploy.
